management-talk — รับ engineer update เข้า แล้วปั้นออกเป็น "ภาษาผู้บริหาร" ตาม channel

กด ถัดไป → ดูทีละ step: text ของ engineer วิ่งผ่าน 3 กฎ (KEEP / STRIP / TRANSLATE) → ได้ core เดียว → ปั้นรูปตาม channel. ตัวอย่างจริงคือ Bug #6084.

Step 0 / 0 เริ่มต้น
💡 คำถามสำคัญ — หยุดคิดก่อนกดต่อ
เรามี core เดียว (ข้อเท็จจริงระดับผู้บริหาร) แล้ว. แต่ Slack ping, standup line, และ exec email ไม่ใช่รูปแบบเดียวกัน — เนื้อหาเดิม แต่เปลือกต่างกัน.
อะไรที่ต้องเปลี่ยนระหว่าง channel? (ความยาว? bolded labels? มี TL;DR ไหม? ทักทาย/ลงท้าย?) กดต่อเพื่อดู core เดียวกันถูกปั้นเป็น 3 channel.
⚠️ ถ้า "แค่ตัดให้สั้น" โดยไม่มีกฎ — มันพังยังไง
  • Strip ไม่พอ: ทิ้ง QuoteDetailService.cs / 02452af ไว้ → ผู้บริหารเอาไปทำอะไรต่อไม่ได้ เป็น noise.
  • Translate มากไป (over-strip): เปลี่ยน "race condition" เป็น "timing issue", "regression" เป็น "small problem" → ดูถูกคนอ่าน และบิดความจริง. กฎคือ แปลโดยไม่โกหก.
  • มั่ว owner: เดาเจ้าของงานจาก git blame → ผิดคน. กฎคือ อย่าแต่ง owner เอง ถ้าไม่รู้ให้ถาม.
  • เลื่อนระดับข้อเท็จจริง: "น่าจะแก้ได้" กลายเป็น "แก้แล้ว" เพื่อให้เล่าสวย → ห้ามเด็ดขาด.
📐 สรุป — management-talk ทำอะไร
1 diagnosis → 3 กฎ → core เดียว → ปั้นตาม channel.
  • KEEP product names, JIRA/ADO keys, PR numbers — สะพานให้ผู้บริหาร cross-reference.
  • STRIP function names, file paths, SHAs, struct fields — ไม่ actionable.
  • TRANSLATE กลไก → cause-and-effect ภาษาคน — โดยไม่ลดทอนศัพท์ระดับ concept (race, regression) และไม่โกหก.
  • Channel shape: JIRA = บล็อกเต็มมี label · Slack = TL;DR + bullets สั้น · standup = บรรทัดเดียว · email = subject + ย่อหน้า · meeting = bullets พูด.
Trade-off / ขอบเขต: มันคือ status update ไม่ใช่ข้อเสนอแนะ · ไม่แต่งข้อเท็จจริง/owner · print-only เป็นค่าเริ่มต้น (โพสต์ขึ้น JIRA เฉพาะเมื่อได้ OK; ไม่โพสต์ Slack/email ให้เอง).
